# i915 Meteor Lake s2idle Resume Regression + MTL-on-Linux Landscape — Research Report

*Generated: 2026-06-20 · Method: 2 parallel web-research agents (WebSearch/WebFetch + `gh api` git ground-truth) · Sources: ~20 · Confidence: HIGH on cause/cure, MEDIUM on exact fix commit*

> Hardware: ThinkPad P1 Gen 7 (Core Ultra 9 185H, Meteor Lake-P, iGPU `7d55`, eDP panel, NVIDIA hybrid). Host: Bazzite / Universal Blue bootc atomic, `ghcr.io/ublue-os/bazzite-nvidia:stable` → kernel `7.0.9-ogc3.2`.

---

## Executive Summary

1. **The bug is real, active, and has no fix on any 7.x kernel.** On s2idle resume the Meteor Lake cx0 C10 PHY DPLL is restored at its parked idle clock (~61.44 MHz) instead of 810 MHz HBR3, tripping `intel_dpll_state_verify` → ~30s `flip_done`/`commit wait` storm and sluggish/black display.
2. **Root cause — CORRECTED.** It is **not** the `fc9be0a10ca4` powerdown-timeout change (that was already fixed pre-6.19 and is present in your kernel). The real regression is the **"Add MTL+ platforms to support dpll framework" series** (lead `1a7fad2aea74`), which is in 7.0, **not** in 6.19, and **still unreverted in mainline master**. Intel is fixing forward.
3. **Only confirmed cure: kernel 6.19.x** (legacy cx0 path). Two independent confirmations. No 7.x kernel — stable, mainline, CachyOS, or -ogc — is confirmed to fix it.
4. **All non-downgrade workarounds fail for this bug:** PSR/DC/FBC kargs (ineffective), the `xe` driver (shares the same cx0 display code; regressed *worse* on Arrow Lake), runtime-PM disable (wrong code path), resume modeset hooks (no confirmed success).
5. **"Stay on 7.0 and fix it" is not achievable today.** The realistic choices are: pin back to a 6.19.x Bazzite image (clean, keeps NVIDIA + Secure Boot), gamble on mainline 7.1 (breaks NVIDIA akmod + Secure Boot, fix unconfirmed), or ride the cosmetic ~30s wake until upstream fixes the framework path.

## PART A — The cx0 Resume Bug: Cause, Fix Status, and Resolution Paths

### A1. The corrected root cause (git ground-truth, VERIFIED)

The previously-tracked theory ("timeout shortened in `fc9be0a10ca4`; fix = its revert / `_MS` patch; not yet in 7.0.9") is **factually wrong on the merge facts**:

- The timeout regression `fc9be0a10ca4` (Nov 1 2025) and its fix `50101556349d9bd0128e65a852d338aca09d6f9b` ("drm/i915/cx0: Replace …_TIMEOUT_US with …_TIMEOUT_MS", Nov 7 2025) landed **6 days apart, both before v6.19**. Source-file ground-truth of `intel_cx0_phy_regs.h`: v6.18 = `…_TIMEOUT_US 100`; **v6.19, v7.0, v7.1 all = `…_TIMEOUT_MS 2`**. The fix has **no `Fixes:` line, no `Cc: stable`** (framed as cleanup).
  - https://github.com/torvalds/linux/commit/50101556349d9bd0128e65a852d338aca09d6f9b
  - https://www.mail-archive.com/intel-gfx@lists.freedesktop.org/msg368675.html
- ∴ the 2 ms timeout is **already in 7.0.9** and **also in the working 6.19** → not the differentiator.

**The real 6.19→7.0 regression is the DPLL-framework conversion** — series *"drm/i915/cx0: Add MTL+ platforms to support dpll framework"*, lead commit **`1a7fad2aea74` "Enable dpll framework for MTL+"** (+ `d174cfb51dce`, `ac3423721117`, et al.). Merge-base tests confirm this series is **in v7.0, NOT in v6.19**, and `1a7fad2aea74` is **still in master, not reverted**. It matches the symptom exactly (new DPLL readout/restore mis-handles the C10 PLL on resume → `mismatch in dpll_hw_state`, `port_clock expected 810000 found 61440`).
  - Series: https://www.mail-archive.com/intel-gfx@lists.freedesktop.org/msg369513.html
  - *VERIFIED:* series is 7.0-only and unreverted. *INFERRED (strong):* it is the cause — matches symptom + the 6.19/7.0 boundary precisely.

### A2. Upstream fix status (VERIFIED: none identifiable)

No upstream commit explicitly fixing the DPLL-framework resume regression exists in any tree as of June 2026. Tracking issues (freedesktop GitLab, titles Anubis-blocked → UNVERIFIED): drm/i915/kernel **#16098**, **#16042**.
- A claim that "a fix landed in 7.1-rc1, commit `afe3f7471623`" is a **misattribution** — that commit is *"drm/i915/dp_tunnel: Don't update tunnel state during system resume"*, unrelated to a plain eDP panel. https://github.com/torvalds/linux/commit/afe3f7471623

### A3. Stable backport (VERIFIED: n/a)

The `_MS` timeout patch has no `Cc: stable`/`Fixes:` and is irrelevant (already present). No stable-tagged fix for the real regression exists — stable cannot backport a fix that isn't upstream yet. No 7.0.10/7.0.11 fix found.

### A5. Atomic-host kernel-swap mechanics (and why they hurt on bazzite-nvidia)

Supported live override (https://coreos.github.io/rpm-ostree/ex-replace/):
```
sudo rpm-ostree override replace --experimental --freeze \
  --from repo='copr:copr.fedorainfracloud.org:group_kernel-vanilla:mainline' \
  kernel kernel-core kernel-modules kernel-modules-core kernel-modules-extra
# undo: rpm-ostree override reset --all   (or rpm-ostree rollback)
```
Two hard blockers on `bazzite-nvidia`:
1. **NVIDIA akmod matching** — ublue bakes a precompiled `kmod-nvidia` pinned to the exact -ogc kernel; any swapped kernel → NVIDIA won't load / deps may fail. https://github.com/ublue-os/akmods
2. **Secure Boot** — ublue signs kernel+akmods with the enrolled Universal Blue MOK; a third-party COPR kernel is unsigned → won't boot under enforcing SB unless self-signed or SB disabled. https://docs.bazzite.gg/General/Installation_Guide/secure_boot/

The clean permanent path is to **rebase to a different bazzite image tag** (kernel + NVIDIA + signing all matched), not a host-side kernel override.

### A6. Non-downgrade workarounds (VERIFIED weak/ineffective)

- **PSR/DC/FBC kargs** — ineffective for this bug (already deployed on the host, still fires). PSR is system-specific: `enable_psr=2` helps some MTL boxes, `enable_psr=0` *causes* freezes on others.
- **`xe` driver** (`i915.force_probe=!7d55 xe.force_probe=7d55`) — won't help: `i915/display/` (incl. `intel_cx0_phy.c`) is **shared/compiled into both** drivers; on Arrow Lake `xe.force_probe` produced an engine-reset storm + GPU wedge (worse). https://www.kernel.org/doc/html/v6.8/gpu/rfc/xe.html · https://www.mail-archive.com/ubuntu-bugs@lists.ubuntu.com/msg6272941.html
- **Disable i915 runtime PM** (`power/control=on`) — wrong code path (this is system-suspend, not runtime PM). No evidence.
- **Resume modeset/DPMS hook** (systemd-sleep post, `kscreen-doctor` off/on) — **no confirmed working report**; failure is in-kernel before userspace gets a usable pipe. Only untested lever, low odds.

### A7. Community resolutions (VERIFIED)

Only confirmed cure anywhere = **downgrade to 6.19.x**: Arch BBS (P1 Gen7 MTL / Gen8 ARL: broken 7.0.3–7.0.4, works 6.19.14); omarchy #5695 (P1 Gen7 155H: broken 7.0.3, works 6.19.13); Ubuntu Bug 2150605 (ARL-S: no fix, PSR/DC/FBC ineffective).
- https://bbs.archlinux.org/viewtopic.php?pid=2297604 · https://github.com/basecamp/omarchy/issues/5695 · https://www.mail-archive.com/ubuntu-bugs@lists.ubuntu.com/msg6272941.html

### A8. Ranked actionable paths

1. **Rebase/pin to a 6.19.x-ogc Bazzite image — Confidence HIGH.** Only evidence-backed cure; keeps NVIDIA akmods + Secure Boot intact (ublue-signed, matched). E.g. rebase to `ghcr.io/ublue-os/bazzite-nvidia:44.20260429` (`6.19.11-ogc1`) and pin/lock the deployment so it doesn't roll forward into 7.0. Trade-off: forgo ~6 weeks of updates + the 7.0 kernel.
2. **Test mainline 7.1 via `@kernel-vanilla` override (or await 7.1-ogc) — Confidence LOW-MEDIUM.** cx0 code differs from 7.0 so *might* help; unconfirmed; host-side override breaks NVIDIA-matching + Secure Boot (self-sign or disable SB). Lowest-friction variant: wait for -ogc to bump to 7.1 (signed, matched) and re-test.
3. **Track upstream + wait — Confidence HIGH it's the correct long-term path, no ETA.** Framework regression unreverted in master; tracked at freedesktop drm/i915/kernel #16098, #16042. Add a P1 Gen7 dmesg.

**Do NOT pursue:** xe switch, runtime-PM disable, PSR/DC/FBC kargs, the `_MS` timeout patch (already present), CachyOS (same bug + no NVIDIA). The `pcie_port_pm=off` experiment is also disfavored — the cause is DPLL-framework state restore, not PCIe refclk PM.

## PART B — Meteor Lake on Linux: Display + Suspend/Resume Landscape

### B1. i915 vs xe on MTL
- MTL i915 declared **stable / default since Linux 6.7** (force_probe dropped). MTL is the **last iGPU gen defaulting to i915**; Lunar Lake/Battlemage default to `xe`. https://www.phoronix.com/news/Linux-6.7-Intel-Meteor-Lake-Gfx · https://www.phoronix.com/review/intel-mtl-i915-xe-linux
- `xe` supports MTL only behind `force_probe`; effectively experimental there (hibernate-resume panics, simpledrm fallback). Display code is **shared** between i915 and xe. https://www.kernel.org/doc/html/v6.8/gpu/rfc/xe.html
- *FLAG (UNVERIFIED):* community claims Fedora 43 (≥6.17) auto-loads `xe` for MTL — likely distro/packaging-specific, not the mainline default. https://raw.githubusercontent.com/spxak1/weywot/main/guides/i915-meteor-6.17.md

### B2. Suspend/resume reality
- **MTL is s2idle/S0ix-only — no S3 "deep."** Never set `mem_sleep_default=deep`. https://blog.fsck.com/agent-blog/2026/03/30/linux-power-tuning-meteor-lake/
- S0ix is fragile (LTR blockers: GBE/ME/IOE; keep `e1000e` loaded; disable AMT/PCH-LAN in BIOS). https://www.feeditout.com/s0ix-sleep-and-hibernate-on-meteor-lake-nvidia-and-the-thinkpad-p1-gen-7/
- The 7.0/7.1 C10 PHY DPLL resume regression reproduces on P1 Gen7/8, Schenker NEO E25, HP ZBook Fury G1i, NVIDIA-independent.
- Separate older hang class: GPU IRQs pinned to one LP-E core + deep C-states → GuC timeouts (6.6 LTS–7.0-rc). https://bbs.archlinux.org/viewtopic.php?id=308313

### B3. Firmware
- Current: DMC `mtl_dmc.bin` v2.23, GuC `mtl_guc_70.bin` 70.53.0, HuC `mtl_huc_gsc.bin` 8.5.4, GSC `mtl_gsc_1.bin`. The resume bug is **firmware-independent** (in-kernel display code).
- `GSC proxy ... didn't bind within the expected timeout` is **mostly cosmetic noise**; real freezes trace to the C10 PHY regression or the IRQ/C-state class. https://bbs.archlinux.org/viewtopic.php?id=308313

### B4. Known-good vs known-bad kernels
- ≤6.6 experimental → 6.7 stable. **6.19.x (6.19.13/14) = known-good** for MTL s2idle resume. **7.0.3–7.0.4 / Ubuntu 7.0.0-14 = known-bad.** 7.1: regression reported still present; fix-landed status **UNVERIFIED**.

### B5. Distro guidance
- Arch: documents i915↔xe switch; consensus = downgrade 6.19.x for the 7.0 regression. Fedora-atomic: kernel-pin/rollback is the practical lever. Lenovo P1 Gen7 = Ubuntu 22.04-certified. Power-tuning vs stability camps disagree on PSR (system-specific). https://wiki.archlinux.org/title/Intel_graphics · https://ubuntu.com/certified/202406-34125/22.04%20LTS

### B6. Trajectory
Converging but **not converged**. Core display/render mature since 6.7; **suspend/resume + S0ix power is the live problem area**, and a 7.x display-resume regression makes "newest kernel" *worse* for MTL right now. Long-term driver direction is `xe` (newer silicon); MTL stays i915.

---

## Key Takeaways / Decision

- **There is no way to keep kernel 7.0 *and* fix this bug today.** Pick one: lose 7.0 (downgrade to 6.19.x — the only confirmed cure) or keep 7.0 (live with the ~30s cosmetic wake / wait for upstream).
- **Cleanest cure on this exact host:** rebase to a 6.19.x-ogc bazzite image and pin it — preserves NVIDIA akmods + Secure Boot. Everything else (COPR/mainline/CachyOS) breaks signing or NVIDIA, or doesn't fix it.
- **The watcher must be repointed** from the (wrong) `fc9be0a10ca4` signature to the DPLL-framework root cause (`1a7fad2aea74`) — done alongside this report.
- **Stop the karg/driver experiments** (`pcie_port_pm`, xe, runtime-PM) — the evidence rules them out.
